What Changed?

Seriously, what changed?

  • Router Facelift (rc.6)
  • Ember Testing (rc.6)
  • Ember Components (rc.6)
  • New actions hash in Router, Controller, View (rc.8)
  • Array based computed properties
  • Preparation for Modules
  • Observers

  • Performace Improvements
  • Larger Documentation
  • Ember Data 1.0 Beta 2

Router Facelift

  • promise-based
  • new model hooks
    • beforeModel
    • model
    • afterModel
  • redirect deprecated
  • loading route
  • error action
  • loading class on links with the {{link-to}} helper
  • gist from @machty

Ember Testing

  • Integration Testing framework
  • no complex drivers (c.f. selenium, headless browsers)
  • uses promises to manange asychronicity
  • built-in adapter for qunit
  • Simple API:

            visit(url)
            find(selector,context)
            fillIn(input_selector, text)
            click(selector)
            keyDown(selector, type, keyCode)
            wait()

Ember Components

  • Views on steroids
  • Ember Components follow the W3C Custom Element spec
  • Define them like templates under components/name
  • They have a different scope, you have to bind properties to them
  • Html content can be rendered inside a component using the {{yield}} helper and the component's block form {{#component-name}}
  • All components are instances of Ember.Component or its subclasses
  • Customizing components is similar to views, you can define tagName, className, custom properties, etc.
  • Actions do not bubble
  • To send action to the application use the sendAction() method
  • TextField and TextArea are now components
  • great tutorial on creating a custom select component

New actions hash

  • events hash in routes deprecated (use actions hash instead)
  • functions on controllers in action dispatch chain deprecated (use actions hash instead)
  • i.e. need to call controller.send(actionName) instead of controller.actionName()

ArrayComputed Example

(stolen from the ember blog)

App.Person = Ember.Object.extend({
    childAges: function() {
        return this.get('children').mapProperty('age');
    }.property('children.@each.age')
});
becomes 

App.Person = Ember.Object.extend({
    childAges: Ember.computed.mapBy('children', 'age')
});
  • When an element is added or removed, the computation is only done once.
  • i.e. if a child is added, their age is appended to childAges

New observer semantics

  • observers do not fire for properties set on init
    • App.Something = Em.Object.extend({
  init: function() {
    // this used to cause the observer to fire
    this.set('someProperty', 'foo');
  },

  somePropertyObserver: function() {
    console.log('someProperty changed');
  }.observes('someProperty').on('init'); // <- now required
});

New observer semantics II

  • observers on computed path WILL NOT FIRE UNTIL PATH IS FORCED
    • App.SomethingController = Em.Controller.extend({
  needs: ['somethingElse']
  
  somethingElse: Em.computed('controller.somethingElse.dongles')

  // will never fire until @get('somethingElse.dongles') is called
  somethingElseDonglesObserver: function() {
    console.log('someProperty changed');
  }.observes('somethingElse.dongles.@each')
});
  • observer use should be rare--almost everything should be done with CPs

Comprehensive Documentation (1.0.0)

EMBER DATA 1.0 BETA 2

  • gigantic release
  • major internals rewrite
  • lazy materialization gone
  • promises used more consistently
  • more consistent API behavior
  • lots of breaking changes
  • read the TRANSITION.md guide
  • port new/small apps now
  • port larger/established apps once things stabilize

Ember Data - Serialization

  • no transformations by default
  • no longer works with ActiveModelSerializers out of the box
  • (possibly) easier for non-rails backends (closer to Django APIs)
  • will eventually be `DS.AMSSerializer`, `DS.DjangoSerializer`, etc.
  • for now, customize (lots of hooks available)

Ember Data - API shift

  • find, all, createRecord, etc. called from store, not model classes
  • store injected into relevant contexts (controllers, routes)

Ember Data - Promises

  • finders used to return "promisey" models
  • i.e. could be used as promises or models that would have data populated async
  • *all* finders now return promises, not models
  • more or less equivalent for success case in model load via route
  • helps clean up custom error handling in routes
  • promise proxies can be bound into templates and data will update on resolve
  • example
